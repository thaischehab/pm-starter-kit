PM Orientado a Dados na Era da IA

Como usar dados e IA para tomar decisões de produto — não apenas para reportar métricas.

O problema com "ser orientado a dados"

Todo PM diz que é orientado a dados. Mas existe uma diferença enorme entre reportar dados e tomar decisões com dados. A maioria faz o primeiro. Poucos fazem o segundo.

Reportar dados é olhar para o dashboard e descrever o que aconteceu. Tomar decisões com dados é usar análise para reduzir a incerteza de uma escolha específica — e saber quando os dados não são suficientes para decidir e você precisa de pesquisa qualitativa.

A armadilha do data-driven: dados descrevem o passado. Produto é sobre o futuro. Usar dados para justificar uma decisão já tomada não é ser orientado a dados — é cherry-picking. Dados reduzem incerteza; não eliminam julgamento.

Os 4 níveis de maturidade em dados para PMs

Nível 1Reativo — "o que aconteceu?"
Comportamento
Lê o dashboard quando algo dá errado. Usa dados para investigar problemas depois que ocorrem.
Ferramentas
Google Analytics, Mixpanel básico, relatórios de CS
Limitação
Não usa dados para antecipar problemas ou guiar decisões proativas. Produto no piloto automático.
Sinal
"A métrica caiu 20% essa semana" — investigação começa depois do fato.
Nível 2Descritivo — "o que está acontecendo?"
Comportamento
Monitora métricas regularmente. Cria dashboards. Reporta para stakeholders com contexto.
Ferramentas
Amplitude, Mixpanel, Tableau, Metabase
Limitação
Foca em descrever comportamento, não em entender causas. Confunde correlação com causalidade.
Sinal
"Nossa retenção é 72% — acima da média do setor." Mas não sabe por que 28% saem.
Nível 3Diagnóstico — "por que está acontecendo?"
Comportamento
Segmenta usuários para entender diferenças de comportamento. Combina dados quantitativos com pesquisa qualitativa. Forma e testa hipóteses.
Ferramentas
Análise de cohort, funil segmentado, entrevistas de churn, A/B testing
Limitação
Ainda é reativo — espera o problema aparecer para investigar.
Sinal
"Usuários que não completam o onboarding têm 3x mais churn. Investigando por quê."
Nível 4Preditivo — "o que vai acontecer e o que fazer?"
Comportamento
Usa padrões de comportamento para antecipar churn, oportunidades e riscos. IA amplifica essa capacidade. Age antes do problema se manifestar.
Ferramentas
Modelos preditivos, análise de behavioral signals, IA para identificação de padrões, alertas proativos
Limitação
Requer dados históricos suficientes e integração entre sistemas. Não elimina a necessidade de julgamento humano.
Sinal
"Usuários com esse padrão de comportamento têm 70% de probabilidade de churnar nos próximos 30 dias. Ativando intervenção proativa."
🤖IA na prática — subir do nível 2 para o nível 3
Use IA para identificar padrões de comportamento que separam usuários de sucesso dos que churnam. Descreva os dados que você tem (métricas de uso, cohorts, funil) e peça para a IA formular hipóteses de por que grupos diferentes se comportam de forma diferente. O output é uma lista de hipóteses para testar — com pesquisa qualitativa e experimentos.
Prompt — diagnosticar diferenças de comportamento entre segmentos
Você é um analista de produto sênior. Tenho os seguintes dados sobre meu produto: - Métrica de retenção em 30 dias: [X%] - Segmentação que identifiquei: [ex: usuários que completam o onboarding têm retenção 2x maior] - Outros padrões observados: [descreva] Contexto do produto: [produto, segmento, estágio] Por favor: 1. Formule 5-7 hipóteses sobre POR QUE esses segmentos se comportam diferente 2. Para cada hipótese, indique: como eu validaria (dados que precisaria ou pesquisa qualitativa) 3. Identifique qual dado crítico está faltando para entender a causa raiz 4. Sugira 3 experimentos ordenados por facilidade de execução vs potencial de aprendizado 5. Aponte onde posso estar confundindo correlação com causalidade

O framework de decisão baseada em dados

Antes de usar qualquer dado para tomar uma decisão, responda essas 5 perguntas:

1. Qual decisão?
Seja específico. "Melhorar retenção" não é uma decisão — "priorizar onboarding ou notificações no próximo sprint" é. Dados sem decisão a tomar são relatórios, não análise.
2. Que dados reduzem a incerteza?
Para cada opção, que evidência mudaria sua decisão? Se não existe dado que te faria mudar de ideia, você já decidiu — pare de coletar dados e decida.
3. Os dados são confiáveis?
Qual é a origem? Há viés de seleção? O período de análise é representativo? Dados de baixa qualidade usados com alta confiança são piores que não ter dados.
4. O que os dados não respondem?
Dados quantitativos dizem O QUÊ mas raramente dizem POR QUÊ. Se a decisão depende de entender motivação, comportamento ou contexto, você precisa de pesquisa qualitativa também.
5. Qual o custo de estar errado?
Se o custo é alto e reversível, invista mais em validação. Se é baixo e reversível, decida rápido e meça. Nem toda decisão merece a mesma profundidade de análise.

Métricas que todo PM precisa dominar

Aquisição
CAC
Custo de Aquisição por Cliente. Se o CAC for maior que o LTV, o negócio está destruindo valor. Todo PM deve conhecer o CAC do seu produto — e como suas decisões o afetam.
Taxa de conversão por canal
Usuarios que vieram de qual canal convertem melhor? Isso informa tanto o produto (onboarding diferenciado) quanto o negócio (onde investir em aquisição).
Ativação e onboarding
TTV (Time to Value)
Tempo até o usuário experimentar o valor central do produto pela primeira vez. Reduzir o TTV é uma das alavancas mais poderosas de retenção — e das mais ignoradas.
Taxa de conclusão do onboarding
Quantos usuários completam as ações críticas iniciais? Segmente por cohort, canal de aquisição e perfil de empresa para entender onde está o gargalo.
Engajamento e retenção
DAU/MAU
Frequência de uso. Um DAU/MAU de 0.5 significa que o usuário médio usa o produto 15 dias por mês. Não compare com benchmarks genéricos — compare com o que faz sentido para o caso de uso do seu produto.
Retenção por cohort
A curva de retenção mais importante é a por cohort — ela mostra se o produto está melhorando ao longo do tempo. Se cohorts mais recentes têm retenção maior, você está evoluindo.
Feature adoption rate
Quantos usuários ativos usam uma feature específica? Features com baixa adoção podem indicar problema de discoverability, UX ruim ou simplesmente que o problema não era real.
Negócio
MRR e expansão
Monthly Recurring Revenue. Mais importante que o MRR total é a composição: quanto vem de novos clientes, expansão de existentes e quanto está sendo perdido em churn.
NPS por segmento
NPS agregado é pouco útil. NPS segmentado por cohort, tamanho de empresa ou feature utilizada revela onde o produto está criando e destruindo valor.
🤖IA na prática — definir as métricas certas para uma nova feature
Use IA para identificar quais métricas realmente capturam o valor que uma feature entrega — e quais são métricas de vaidade que parecem importantes mas não informam decisões. Descreva a feature e o usuário: a IA ajuda a definir métrica primária, secundária e guardrail antes do desenvolvimento começar.
Prompt — definir métricas de sucesso para uma feature
Vou lançar a seguinte feature: [descreva a feature e o problema que resolve] Usuário-alvo: [perfil do usuário] Hipótese central: [se fizermos X, então Y acontecerá] Métrica North Star do produto: [qual é a métrica mais importante do produto] Por favor: 1. Sugira 1 métrica primária de sucesso — deve conectar diretamente com a hipótese 2. Sugira 2 métricas secundárias — sinais que confirmariam o impacto 3. Sugira 1-2 métricas guardrail — o que não pode piorar 4. Identifique métricas de vaidade que eu poderia confundir com sucesso real 5. Defina o que seria um resultado positivo, neutro e negativo após 30/60/90 dias 6. Aponte os riscos de medir errado essa feature específica
Regra prática: se você não consegue explicar em uma frase como uma métrica conecta ao valor do usuário ou ao resultado do negócio, provavelmente é uma métrica de vaidade. Pageviews, número de features entregues e "usuários cadastrados" raramente informam decisões de produto.
