User Story com Critérios de Aceite

Escreva histórias que o time de engenharia realmente entende — com o formato certo e exemplos reais.

O que é uma User Story

Uma user story descreve uma funcionalidade do ponto de vista do usuário: "Como [tipo de usuário], quero [ação], para que [benefício]." Os critérios de aceite definem com precisão quando essa story está concluída e testada.

Erro mais comum: escrever stories do ponto de vista do sistema. "O sistema deve enviar um email" não é user story — é requisito técnico. Escreva sempre da perspectiva de quem usa o produto. Se você não consegue identificar o usuário, é sinal de que a story ainda não está madura.
🤖IA na prática
Use IA para gerar user stories a partir dos requisitos do PRD. Cole os requisitos funcionais e o contexto do usuário. A IA quebra em stories no formato correto com critérios de aceite estruturados. Você revisa, ajusta o que não reflete o contexto real e complementa com casos de borda que só você conhece.
Prompt — gerar user stories de requisitos
Converta os requisitos funcionais abaixo em user stories com critérios de aceite. Contexto: - Produto: [descrição] - Persona principal: [nome/perfil do usuário] - Requisitos funcionais: [cole os RFs] Para cada story: 1. Escreva no formato "Como [usuário], quero [ação], para que [benefício]" 2. Adicione 2-3 critérios de aceite no formato Given-When-Then 3. Defina o que está fora do escopo desta story específica 4. Estime a complexidade (S/M/L) com justificativa Identifique também: quais stories têm dependência entre si e qual a ordem de implementação recomendada.

Formato Given-When-Then

Given (dado que)
O contexto inicial — estado do sistema e do usuário antes da ação acontecer
When (quando)
A ação específica que o usuário ou o sistema executa naquele momento
Then (então)
O resultado observável e verificável que deve acontecer imediatamente após a ação

Exemplos preenchidos

Story 1Receber alerta de vencimento de contrato
Story
Como gestora de contratos, quero receber um alerta 7 dias antes do vencimento, para que eu tenha tempo de acionar o fornecedor antes que o prazo expire.
Critério 1
Given que tenho um contrato com vencimento em 7 dias e notificações ativas
When o sistema processa os alertas diários às 8h no meu horário
Then recebo um email com: nome do contrato, data de vencimento, dias restantes e link direto para o contrato
Critério 2
Given que o contrato não foi renovado e faltam 24 horas
When o sistema processa os alertas
Then recebo um alerta adicional com urgência visual no assunto do email ("⚠️ Vence amanhã: [nome]")
Critério 3
Given que desativei notificações para um contrato específico
When esse contrato se aproxima do vencimento
Then não recebo alertas desse contrato, mas continuo recebendo normalmente dos demais
Fora do escopo
Notificação por SMS. Renovação automática. Notificação para outros membros do time. Alertas para contratos já vencidos.
Complexidade
M — lógica de agendamento conhecida, dependência: Sendgrid configurado, template aprovado
Story 2Configurar antecedência dos alertas por contrato
Story
Como gestora de contratos, quero configurar com quantos dias de antecedência recebo alertas de cada contrato, para ajustar de acordo com a criticidade e o tempo de negociação necessário.
Critério 1
Given que estou nas configurações de um contrato específico
When acesso a opção "Configurar alertas"
Then vejo as opções de 7, 15 e 30 dias com checkboxes (posso selecionar múltiplos)
Critério 2
Given que alterei a configuração de alertas de um contrato
When salvo as alterações
Then vejo uma confirmação visual e os próximos alertas respeitam a nova configuração
Critério 3
Given que um contrato não tem configuração personalizada
When o sistema processa alertas
Then usa a configuração padrão da conta (7 dias), sem gerar erro
Complexidade
S — configuração simples, sem dependências externas
🤖IA na prática
Use IA para identificar casos de borda que você esqueceu. Cole suas user stories e critérios de aceite e peça para a IA simular um testador tentando quebrar a feature. Ela vai sugerir cenários de erro, estados inválidos e comportamentos inesperados que você provavelmente não cobriu.
Prompt — encontrar casos de borda
Atue como um QA sênior tentando quebrar as user stories abaixo. Stories e critérios de aceite: [cole as stories] Identifique: 1. Casos de borda não cobertos pelos critérios de aceite 2. Estados inválidos ou inesperados do sistema (ex: o que acontece se o usuário não tiver email?) 3. Condições de corrida ou problemas de concorrência 4. Cenários de erro que precisam de tratamento explícito 5. Critérios de aceite ambíguos que podem ser interpretados de formas diferentes por devs diferentes Para cada problema encontrado, sugira como adicionar ou corrigir o critério de aceite correspondente.
Dica de PM sênior: antes de escrever a story, pergunte: qual problema essa story resolve? Se não conseguir responder em uma frase, volte para o discovery. Story sem problema claro vira feature órfã que ninguém usa — e ninguém vai te avisar que ninguém usou.
